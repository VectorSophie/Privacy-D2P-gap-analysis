# 설계 결정 기록 (Architecture Decision Records)

각 항목은 **컨텍스트 → 결정 → 근거 → 트레이드오프** 형식으로 기술합니다.

---

## ADR-001. 파이프라인을 8개의 독립 단계로 분리

**컨텍스트**  
전체 분석은 수집 → 크롤링 → 추출 → 트래커 탐지 → LLM 평가 → Mismatch 산출 → 통계 → 보고서 순서로 진행된다. 각 단계는 실행 시간과 비용이 크게 다르다(크롤링 수십 분, LLM 평가 비용 발생).

**결정**  
각 단계를 독립 CLI 명령어로 분리하고, 중간 산출물을 파일(`data/interim/`, `data/processed/`)로 저장한다.

**근거**  
- 특정 단계가 실패하거나 파라미터를 바꿔 재실행할 때 앞 단계를 반복하지 않아도 된다.
- LLM 평가처럼 비용이 발생하는 단계는 결과를 캐싱하여 불필요한 API 호출을 방지한다.
- 연구자가 중간 결과물을 직접 검사하고 디버깅할 수 있다.

**트레이드오프**  
단계 간 스키마 변경 시 중간 파일을 수동으로 무효화해야 한다. 일관성 보장을 위해 각 단계는 항상 이전 단계의 출력 파일을 덮어쓰는 방식으로 설계되었다.

## ADR-002. LLM 출력을 Pydantic 스키마로 강제

**컨텍스트**  
GPT-4는 `response_format: json_object`를 지정해도 스키마 자체를 반환하거나, 필드를 누락하거나, 설명 텍스트를 섞는 경우가 있다.

**결정**  
`ComplianceEvaluation` Pydantic v2 모델로 출력 형식을 정의하고, `model_validate_json()`으로 파싱한다. 파싱 실패 시 `MockPolicyEvaluator`로 폴백한다.

**근거**  
- 526건 전체에 대해 동일한 필드 구조를 보장하여 downstream 집계 코드가 단순해진다.
- 파싱 실패를 silent error로 처리하지 않고 경고 로그를 남겨 오염된 결과를 추적할 수 있다.
- JSON 스키마를 프롬프트에 직접 주입하면 GPT-4가 스키마 객체 자체를 반환하는 오류가 발생한다. 구체적인 출력 예시(`_OUTPUT_EXAMPLE`)를 제공하는 방식으로 해결했다.

**트레이드오프**  
Pydantic 스키마 변경 시 기존 `llm_eval.json` 파일이 역호환되지 않아 재평가가 필요하다.

## ADR-003. Mock / Real LLM 이중 평가기 구조

**컨텍스트**  
개발·테스트 시마다 GPT-4 API를 호출하면 비용이 발생하고 속도가 느리다.

**결정**  
`BasePolicyEvaluator` 추상 클래스를 두고, `MockPolicyEvaluator`(키워드 기반)와 `OpenAIPolicyEvaluator`를 구현한다. `get_evaluator(use_mock=True/False)`로 런타임에 선택한다.

**근거**  
- `use_mock: true`로 설정하면 API 키 없이 파이프라인 전체를 실행하고 결과 형식을 검증할 수 있다.
- Mock 결과는 키워드 매칭 기반이라 정확도는 낮지만, 파이프라인 통합 테스트에는 충분하다.
- 동일한 인터페이스를 공유하므로 호출부 코드를 변경하지 않고 평가기를 교체할 수 있다.

**트레이드오프**  
Mock 결과를 실제 분석에 사용하면 안 된다. `use_mock: false`가 논문 결과의 기준값이다.

## ADR-004. 처리방침 탐색에 BFS 깊이 2 적용

**컨텍스트**  
처리방침 페이지는 홈페이지에서 직접 링크되기도 하고, footer → 약관 → 처리방침처럼 한 단계 더 들어가야 하는 경우도 있다.

**결정**  
루트 도메인에서 시작하여 BFS 깊이 최대 2까지 순회하며 처리방침 페이지를 탐색한다(`max_bfs_depth: 2`).

**근거**  
- 깊이 1(홈페이지만): 처리방침이 footer나 하위 약관 페이지에 있는 경우 누락.
- 깊이 3 이상: 탐색 시간이 기하급수적으로 증가하고 관련 없는 페이지까지 방문.
- 실험적으로 깊이 2가 커버리지와 속도의 적절한 균형점으로 확인되었다.

**트레이드오프**  
일부 기업은 처리방침을 팝업이나 모달로만 제공하거나 별도 서브도메인에 위치시켜, BFS로 탐지되지 않는 경우가 있다.

## ADR-005. 트래커를 도메인 기반 룰셋으로 분류

**컨텍스트**  
네트워크 요청을 트래커 카테고리로 분류하는 방법으로 외부 blocklist(EasyList, Disconnect 등) 사용과 자체 룰셋 작성 두 가지를 검토했다.

**결정**  
`TRACKER_DB` 딕셔너리(도메인 → 카테고리 매핑)를 `tracker.py` 내에 직접 관리한다. 외부 의존성을 두지 않는다.

**근거**  
- 외부 blocklist는 한국 서비스(Naver 분석, Kakao 트래커 등)에 대한 커버리지가 낮다.
- 연구 목적상 분류 기준이 명시적이고 고정되어야 재현성을 보장할 수 있다.
- 룰셋 변경 시 이유와 시점을 코드 히스토리로 추적할 수 있다.

**트레이드오프**  
룰셋에 없는 도메인은 `Unknown Third-party`로 남는다. 초기 탐지 시 Unknown이 44.3%를 차지하였으며, 이는 롱테일 서드파티 서비스의 존재와 분류 규칙의 불완전성을 동시에 반영한다. 1차 보강(한국 광고 네트워크, TikTok, Intercom 등 추가)으로 Unknown 비율 감소 예정이며, `scripts/analyze_unknown_trackers.py`로 잔여 미분류 도메인을 지속 모니터링한다.

## ADR-006. LLM 입력 텍스트를 6,000자로 절단

**컨텍스트**  
GPT-4 Turbo의 컨텍스트 윈도우는 128K 토큰으로 충분하지만, 처리방침 전문 입력 시 입력 비용이 증가하고 불필요한 boilerplate가 포함될 수 있다.

**결정**  
입력 텍스트를 앞에서 6,000자로 절단한다. 출력은 `max_tokens=1024`로 제한한다.

**근거**  
- 처리방침의 핵심 조항(수집 목적, 제3자 제공, 트래커 언급)은 대부분 문서 앞부분에 위치한다.
- 6,000자는 평균적인 한국어 처리방침 전문을 포괄하면서도 API 비용을 통제한다.
- `max_tokens=1024`는 JSON 응답 구조를 완성하기에 충분하며, 미설정 시 응답이 잘려 파싱 오류가 발생하는 사례가 확인되었다.

**트레이드오프**  
처리방침 후반부에만 트래커 언급이 있는 경우 `mentions_third_party_trackers`가 False로 잘못 분류될 수 있다.

## ADR-007. 기업 중복 제거에 SequenceMatcher 사용

**컨텍스트**  
MSME 명단과 수동 큐레이션 목록에 동일 기업이 다른 표기로 등장한다("카카오스타일" vs "카카오스타일(주)").

**결정**  
Python 표준 라이브러리 `difflib.SequenceMatcher`로 정규화된 기업명의 유사도 ≥ 0.88인 경우 중복으로 처리한다. 임계값은 `configs/base.yaml`의 `dedup_name_threshold`로 조정 가능하다.

**근거**  
- 임베딩 기반 유사도는 별도 모델 의존성이 생기고 재현성 보장이 어렵다.
- 레벤슈타인 거리는 한국어 법인 접미사 변형에 취약하다.
- SequenceMatcher는 결정론적이며, 법인 접미사 정규화(`_CORP_RE`) 전처리와 결합하면 충분한 정확도를 보인다.

**트레이드오프**  
영문 표기와 한글 표기가 섞인 경우("Kakao Style" vs "카카오스타일")는 탐지되지 않는다. 현재 데이터셋에서는 이런 혼용 사례가 극소수여서 허용 가능한 수준이다.

## ADR-008. asyncio.Semaphore로 동시 크롤링 수 제한

**컨텍스트**  
526개 사이트를 순차 처리하면 수 시간이 소요된다. 무제한 병렬화는 대상 서버에 과도한 부하를 주고 IP 차단 위험이 있다.

**결정**  
`asyncio.Semaphore(5)`로 동시 실행 수를 5로 제한하고, 요청 간격을 `rate_limit_ms: 2000`으로 설정한다.

**근거**  
- 동시성 5는 실제 실행 시 스로틀링 없이 크롤링이 완료되면서 단일 서버에 집중 부하를 주지 않는 경험적 수치다.
- `rate_limit_ms`는 `configs/base.yaml`에서 조정 가능하므로 대상 서버의 민감도에 따라 튜닝할 수 있다.
- `robots.txt` 준수와 rate limiting을 함께 적용하여 윤리적 크롤링 원칙을 준수한다.

**트레이드오프**  
크롤링 전체 완료까지 수십 분이 소요된다. 속도가 중요한 경우 Semaphore 값을 높이되 rate_limit_ms도 함께 조정해야 한다.
